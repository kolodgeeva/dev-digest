## What Works
<!-- approaches & solutions that worked -->
- 2026-06-22 · discovery · Before building a "new" feature on branch L02, GREP FOR SCAFFOLDING FIRST — several features are pre-seeded across layers and meant to be reused, not recreated. The Conventions extractor already had: the `conventions` table in `db/schema/knowledge.ts` (note: NOT a `schema/conventions.ts`) wired into the schema barrel; an (unused) `ConventionCandidate` Zod contract mirrored in both `*/vendor/shared/contracts/knowledge.ts`; a `MockLLMProvider` `structuredBySchema` comment naming the exact two-step dialogue (`ConventionFileSelection`→`ConventionExtraction`); a `conventions` entry in `FeatureModelId`/`FEATURE_MODELS` (`vendor/shared/contracts/platform.ts`); and `repoIntel.getConventionSamples()`. Checking these first turned a planned "new table + migration + new contracts" into zero migrations and zero shared-contract edits. Quick check: `grep -rn "<feature>" server/src/db/schema server/src/vendor/shared/contracts server/src/adapters/mocks.ts`.
- 2026-06-22 · solution · Non-agent LLM features resolve provider+model via `resolveFeatureModel(container, workspaceId, '<featureId>')` (`modules/settings/feature-models.ts`) — workspace Settings override, else the `FEATURE_MODELS` registry default — then `(await container.llm(provider)).completeStructured({ model, schema, schemaName, messages })`. The `<featureId>` must be in the `FeatureModelId` enum (`vendor/shared/contracts/platform.ts`). Do NOT hardcode a model constant; eval/onboarding/brief/conventions all go through this. For a multi-step structured flow, give each call a DISTINCT `schemaName` and have tests inject `new MockLLMProvider('openai', { structuredBySchema: { SchemaA: fixtureA, SchemaB: fixtureB } })` — the mock dispatches by `req.schemaName` and falls back to `structured` (`adapters/mocks.ts`).
- 2026-06-17 · solution · Route IT tests live in `server/test/*.it.test.ts` (NOT under `src/`; `vitest.config.ts` includes `test/**`). Pattern (copy `test/pulls-comments.it.test.ts`): `const d = (await dockerAvailable()) ? describe : describe.skip` to gate on Docker; `startPg()` (`test/helpers/pg.ts`) spins pgvector + runs migrations; `seed(db)` then read the first workspace for its id; build the app per-test with `buildApp({ config, db, overrides: { github: gh } })` and drive it via `app.inject({ method, url })`. For list/detail routes that sync from GitHub on read, pass `new MockGitHubClient({ pulls: [] })` so the sync doesn't overwrite your hand-seeded rows. Example: `test/pulls-findings-summary.it.test.ts`.
- 2026-06-28 · solution · To prove a feature makes ZERO model calls in an IT test, inject `overrides.llm = { openai: new MockLLMProvider('openai'), anthropic: new MockLLMProvider('anthropic') }` PLUS `overrides.secrets = { get: async () => undefined }` (offline → any non-mocked provider build throws ConfigError instead of hitting the network), drive the route via `app.inject`, then assert `mock.calls.length === 0` (`MockLLMProvider.calls` is a public array recording every method call, `adapters/mocks.ts:~95`). NOTE a deterministic READ route like `/pulls/:id/smart-diff` reads `pr_files` straight from the DB and does NOT sync from GitHub (unlike `/pulls/:id`, which deletes+reinserts `pr_files` on every read), so the IT test must SEED `t.prFiles` rows directly. Evidence: `test/smart-diff.it.test.ts`.

## What Doesn't Work
<!-- dead ends & antipatterns — the most-skipped, most-valuable section; fill it -->
- 2026-06-23 · antipattern · An IT test is only hermetic for an LLM call if it injects `overrides.llm[<provider>]` for the EXACT provider the code resolves — `container.llm(id)` returns the mock only when `overrides.llm[id]` is set; otherwise it builds a REAL provider whose key comes from `LocalSecretsProvider` reading `~/.devdigest/secrets.json` (NOT just `process.env`, so a clean shell env doesn't protect you), and the test makes a live network call (symptom: one test runs ~30–40s instead of ~1s and is non-deterministic). This bites when the code has provider FALLBACK logic (`resolveAvailableLlm`): a test exercising fallback injects only the fallback provider's mock, leaving the default provider (openai) to build for real. FIX: also inject `overrides.secrets = { get: async () => undefined }` so every non-mocked provider build throws `ConfigError` deterministically (offline) and resolution lands on your injected mock. Evidence: `test/conventions.it.test.ts` fallback case.

## Codebase Patterns
<!-- conventions, architectural decisions -->
- 2026-06-28 · codebase-pattern · A deterministic, token-FREE PR feature (Smart Diff — `GET /pulls/:id/smart-diff`, `modules/smart-diff/`) composes data already in the DB: `container.reviewRepo.getPrFiles(prId)` for the changed files + `reviewRepo.reviewsForPull(prId)[0].findings` for the LATEST review's findings (the helper returns reviews newest-first, so `[0]` is latest; finding rows carry `startLine` → contract `finding_lines`, which are NEW-side line numbers). The service has NO `llm`/`resolveAvailableLlm` import — that absence IS the "free by tokens" guarantee. The `SmartDiff` Zod contract (`groups[{role, files[]}]` + `split_suggestion`) already existed in `vendor/shared/contracts/brief.ts` (it was pre-scaffolded like the conventions feature — grep brief.ts before adding contracts). Layering mirrors `modules/intent/` but simpler (no diff-load, no LLM): `routes.ts`→`service.ts`→`reviewRepo`, pure `classify.ts` (`buildSmartDiff`) for the domain logic, thresholds/patterns in `constants.ts`. Reference: `modules/smart-diff/{classify,service,routes}.ts`.
- 2026-06-22 · codebase-pattern · When a service needs ANOTHER module's repository (e.g. conventions→skills/agents to create+link a skill), do NOT `new SkillsRepository(db)` or `import { SkillsRepository }` inside the service — `pnpm lint:arch` fails it as `no-cross-module-coupling` (error), and any `import * as t from db/schema` / `import { eq } from 'drizzle-orm'` in a service fails as `app-no-direct-db` (error). FIX: expose the foreign repo as a lazy getter on `platform/container.ts` (the composition root is the ONLY place allowed to import every module's repository — `agentsRepo`/`reviewRepo`/`skillsRepo` already live there as `get x() { return this._x ??= new XRepository(this.db) }`), then call `this.container.skillsRepo.insert(...)`. Any DB lookup the service wants (e.g. repo display name) goes as a method ON the module's OWN repository, not an inline query. The ONLY residual `warn` you can't avoid is importing a sibling's pure `helpers.ts` (e.g. `skills/helpers.toSkillDto` to map the returned row) — `no-cross-module-domain`, accepted precedent (same as `run-executor→skills/helpers`). Reference: `modules/conventions/service.ts`.
- 2026-06-17 · codebase-pattern · Per-run LLM cost is ALREADY computed inside reviewer-core — `ReviewOutcome.costUsd` (`reviewer-core/src/review/run.ts:110,184`; OpenRouter's API `cost`, else `estimateCost` fallback). To surface cost, just persist `outcome.costUsd`; never recompute via the model/pricing again (zero extra calls). `run-executor.ts:213` destructures it; `estimateCost`/`price-book` already wired in the container.
- 2026-06-17 · codebase-pattern · PR-list per-PR aggregates (score, cost_usd) are computed ON READ, not denormalized. `pulls/routes.ts` (~line 120) builds `latestReviewByPr` from the `reviews` table ordered newest-first, first-seen-per-PR wins. cost lives on `agent_runs` (not `reviews`), so it's a `leftJoin(agentRuns, eq(agentRuns.id, reviews.runId))`. Mirror this map to add any new latest-review-derived list field.
- 2026-06-17 · codebase-pattern · CORRECTION of the entry above re cost_usd: PR-list `cost_usd` is NOT the latest-review run cost — it's the SUM of every agent run for that PR (across all agents, all time). Built in `pulls/routes.ts` as a separate `costByPr` map: `select({ prId, cost: sql\`sum(${agentRuns.costUsd})\` }).from(agentRuns).where(inArray(prId, prIds)).groupBy(prId)`. `score` still comes from `latestReviewByPr`. `sum()` over double-precision returns `number|null` (null when all rows unpriced → renders "—"). Two independent queries now, not one join.
- 2026-06-17 · codebase-pattern · PR-list `findings_summary` ({CRITICAL,WARNING,SUGGESTION}|null on PrMeta) is derived from each PR's LATEST review (consistent with `score`), in `pulls/routes.ts`: extend `latestReviewByPr` to also capture the review `id`, invert it to a `reviewId→prId` map, then ONE grouped query `select({ reviewId, severity, count: sql\`count(*)\` }).from(findings).where(inArray(reviewId, latestReviewIds)).groupBy(reviewId, severity)`. Count ALL findings (do NOT exclude `dismissedAt`) so the badge total matches the detail popover's "N findings". Set `findings_summary: findingsByPr.get(r.id) ?? null` — null for unreviewed PRs AND for a latest review with zero findings. `findings.severity` is a free `text` column, so guard the three known values before indexing the counts object.
- 2026-06-21 · codebase-pattern · The backend onion layering is now machine-enforced — run `pnpm lint:arch` (`server/.dependency-cruiser.cjs`) before committing module changes; CI should too. Layer map per `modules/<name>/`: `routes.ts`=presentation (Zod+getContext, calls one service), `service.ts`=application (calls repo+ports, no Drizzle), `types/constants/helpers.ts`=domain (pure, no Fastify/Drizzle/adapters), `repository.ts`/`repository/*.repo.ts`=the ONLY place Drizzle/`db/*` lives. Port INTERFACES live in `@devdigest/shared` (LLMProvider, GitClient…); `adapters/*` implement them; `platform/container.ts` is the composition root (allowed to import everything). ERROR rules (break build): repo→routes/service, service→Drizzle, domain→Fastify/Drizzle/adapters, cross-module `modules/a`→`modules/b`. Reference clean module = `modules/agents/`. Full rationale in `.claude/skills/onion-architecture/`.
- 2026-06-30 · codebase-pattern · A synchronous "create→wait→collect" route (for MCP `run_agent_on_pr`) lives at `POST /reviews/run-sync` (`modules/reviews/routes.ts`): start one run via `service.runReview`, then block on the IN-PROCESS `container.runBus.onDone(runId, …)` with a `setTimeout` race (`waitForDone` helper) capped by `MCP_RUN_TIMEOUT_MS` (default 120s) — NO DB polling. On timeout it returns `{ status:'running', run_id }` so the caller fetches later via `GET /runs/:id/outcome`. The wait MUST live in the API process (not the MCP client) because `runBus` is in-memory and process-local. Rate-limit it the same as the manual trigger (`config: { rateLimit: { max: 10, timeWindow: '1 minute' } }`) — same LLM fan-out cost.
- 2026-06-30 · codebase-pattern · MCP/external callers speak `owner/name` + PR number + run_id, NOT internal ids. Resolution seams added to `ReviewService`: `resolveRepoId` / `resolvePullRef` / `runOutcome` (`modules/reviews/service.ts`), backed by workspace-scoped repo queries `findRepoByFullName(workspaceId, …)`, `findPullByRepoAndNumber(workspaceId, repoId, number)`, `getRunById(workspaceId, runId)` (`repository/pull.repo.ts`, `run.repo.ts`). EVERY new lookup takes `workspaceId` as the first filter — the one exception, `reviewForRun(runId)` (`review.repo.ts`), is unscoped but SAFE only because callers gate it behind a workspace-scoped `getRunById` first. Resolution failures throw `NotFoundError` whose message LEADS ONWARD (e.g. "call list_agents to get a valid agent id") — the MCP client passes it straight to the tool-caller. IT coverage: `test/mcp-http.it.test.ts`.
- 2026-06-21 · what-doesnt-work · Known onion-layer DEBT flagged as `warn` by `pnpm lint:arch` (don't add more; fix when you touch them): DB-in-route — `settings/workspace/polling/pulls` `routes.ts` import `drizzle-orm`+`db/schema` directly instead of going through a service; `repos/service.ts` imports a sibling module's `repo-intel/constants.ts` (cross-module coupling — promote to `_shared`); `repos/helpers.ts` reaches into `db/schema` for a row type (`typeof t.repos.$inferSelect`) — prefer a type from `db/rows.ts`.

## Tool & Library Notes
<!-- dependency quirks/gotchas -->
- 2026-06-21 · tool-note · dependency-cruiser classifies `import * as t from '../../db/schema.js'` as a RUNTIME dependency even when `t` is used only in type position (`typeof t.repos.$inferSelect`), and `to.dependencyTypesNot: ['type-only']` does NOT suppress it (only `import type`/`import { type X }` are tagged type-only). So a layering rule that must tolerate type-only row-shape references can't rely on `dependencyTypesNot` — instead split the rule by target (e.g. `db/schema` at `warn`, `fastify|drizzle-orm|adapters` at `error`). Config: `server/.dependency-cruiser.cjs`; needs `options.tsConfig.fileName: 'tsconfig.json'` to resolve `@devdigest/*` path aliases.

## Recurring Errors & Fixes
<!-- repeated error + its fix -->
- 2026-06-17 · gotcha · Adding a field to a shared Zod contract as `z.number().nullable()` makes the KEY REQUIRED (nullable ≠ optional) — every `Contract.parse({...})` fixture missing it throws "Required". When touching RunStats/RunSummary/etc., grep fixtures and update them: `server/test/contracts.test.ts`, plus client `RunTraceDrawer.test.tsx` / `RunHistory.test.tsx`. Use `.nullish()` instead if the key should be optional (that's what PrMeta.score/cost_usd use).
