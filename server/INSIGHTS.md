## What Doesn't Work

- 2026-06-23 Â· antipattern Â· An IT test is only hermetic for an LLM call if it injects `overrides.llm[<provider>]` for the EXACT provider the code resolves â€” `container.llm(id)` returns the mock only when `overrides.llm[id]` is set; otherwise it builds a REAL provider whose key comes from `LocalSecretsProvider` reading `~/.devdigest/secrets.json` (NOT just `process.env`, so a clean shell env doesn't protect you), and the test makes a live network call (symptom: one test runs ~30â€“40s instead of ~1s and is non-deterministic). This bites when the code has provider FALLBACK logic (`resolveAvailableLlm`): a test exercising fallback injects only the fallback provider's mock, leaving the default provider (openai) to build for real. FIX: also inject `overrides.secrets = { get: async () => undefined }` so every non-mocked provider build throws `ConfigError` deterministically (offline) and resolution lands on your injected mock. Evidence: `test/conventions.it.test.ts` fallback case.

- 2026-06-22 Â· codebase-pattern Â· When a service needs ANOTHER module's repository (e.g. conventionsâ†’skills/agents to create+link a skill), do NOT `new SkillsRepository(db)` or `import { SkillsRepository }` inside the service â€” `pnpm lint:arch` fails it as `no-cross-module-coupling` (error), and any `import * as t from db/schema` / `import { eq } from 'drizzle-orm'` in a service fails as `app-no-direct-db` (error). ## Tool & Library Notes

- 2026-06-21 Â· tool-note Â· dependency-cruiser classifies `import * as t from '../../db/schema.js'` as a RUNTIME dependency even when `t` is used only in type position (`typeof t.repos.$inferSelect`), and `to.dependencyTypesNot: ['type-only']` does NOT suppress it (only `import type`/`import { type X }` are tagged type-only). Config: `server/.dependency-cruiser.cjs`; needs `options.tsConfig.fileName: 'tsconfig.json'` to resolve `@devdigest/*` path aliases.
