## What Works
<!-- approaches & solutions that worked -->
- 2026-06-17 · convention · Money/cost display renders "—" for null (unknown model / failed run), NEVER "$0.00" — a missing price must be visually distinct from a genuine zero. A priced-zero run (free model) shows a real "$0.0000". Single source: `client/src/lib/format-cost.ts` (`formatCost`).
- 2026-06-17 · solution · Hover popovers that live inside the PR-list table must escape the `tableCard`'s `overflow: hidden` (`pulls/styles.ts:~91`) — a `position:absolute` child gets clipped (peeks out one row, then cut off). The reusable `HoverCard` (`client/src/components/hover-card.tsx`) solves this by rendering content through a React portal to `document.body` with `position: fixed`, anchored to the trigger's `getBoundingClientRect()`: computes pos on open, clamps `left` to the viewport, and flips above the trigger when it would overflow the viewport bottom (last-row case). Reach for this (not z-index/overflow hacks on the table) whenever a popover must escape an `overflow:hidden`/scroll ancestor.

## What Doesn't Work
<!-- dead ends & antipatterns — the most-skipped, most-valuable section; fill it -->
- 2026-06-17 · antipattern · A popover that closes on `window` scroll via a CAPTURE-phase listener (`addEventListener('scroll', close, true)`) also fires for scrolls INSIDE the popover — so the moment you try to scroll its own list it self-closes, and the scrollbar looks broken. Guard the handler: `if (popRef.current?.contains(e.target)) return;` — only close on outside/page scroll. `client/src/components/hover-card.tsx`.
- 2026-06-18 · antipattern · A flip-above popover (one that overflows the viewport bottom and re-anchors above its trigger) must NOT be `top`-anchored as `top = triggerTop - GAP - height`: that `top` is computed once for the then-current height, so if the content later SHRINKS (e.g. click-to-filter cuts a 4-item list to 1) the popover stays pinned high and a big gap opens between it and the trigger. Anchor the flipped case by `bottom` instead (`bottom = innerHeight - (triggerTop - GAP)`) so it stays glued just above the trigger and grows/shrinks upward — no recompute-on-resize needed. The below-trigger case is already fine top-anchored. `client/src/components/hover-card.tsx` (`Pos = {left, top|null, bottom|null}`, flip layout-effect is idempotent: only flips/unflips when placement actually differs).
- 2026-06-17 · antipattern · In a fixed-width popover, an unbroken token (e.g. a long `src/a/b/c/File.tsx:129` path) forces a horizontal scrollbar. Don't add `overflowX:auto` — set `overflowX:"hidden"` on the scroll container and let the path wrap with `overflowWrap:"anywhere"` (keep the full value in a `title` tooltip). `client/src/components/findings-popover.tsx`.
- 2026-06-17 · antipattern · Don't bury a formatter used by multiple route segments inside a feature's `_components/.../helpers.ts`. `formatCost` is needed by both the PR list (`pulls/`) and the run trace/timeline (`pulls/[number]/`) — cross-segment relative imports (`../[number]/.../helpers`) are ugly and brittle. Put shared formatters in `client/src/lib/` and import via `@/lib/...`. Single-use ones (formatTokens/formatSeconds) correctly stay local to the trace drawer.

## Codebase Patterns
<!-- conventions, architectural decisions -->
- 2026-06-17 · codebase-pattern · Findings on the PR detail page are NOT one flat list — they live per review run, rendered `FindingsTab` → `ReviewRunAccordion` (one per run) → `FindingsPanel` → `FindingCard`. A PR-wide finding filter therefore threads a prop down all three levels AND applies inside `visibleFindings()` (`_components/FindingsPanel/helpers.ts`), which is the single chokepoint that filters+sorts (compose new filters here so they stack with hide-low-confidence). For the count/summary, aggregate with `runs.flatMap(r => r.findings)` — same source as `allFindings` in `page.tsx`. See the `?severity` filter (`page.tsx`, `_components/FindingsTab/SeverityFilterBar.tsx`).
- 2026-06-17 · codebase-pattern · PR detail page filter/view state goes in the URL via the `setParam(key, val)` helper in `page.tsx` (null deletes the key), matching the existing `?tab` / `?trace`. Validate enum-ish params against the contract enum before use (e.g. `Severity` from `@devdigest/shared`) and coerce unknowns to null — don't trust the raw query string. Survives reload/back-forward/sharing for free.
- 2026-06-17 · codebase-pattern · Adding a PR-list table column = 3 edits that MUST stay in sync (`client/src/app/repos/[repoId]/pulls/`): append the key to `COLUMN_KEYS` (constants.ts), add a matching width to `GRID` (constants.ts — column count must equal width count), and insert the cell in `_components/PRRow/PRRow.tsx` at the SAME ordinal position. The header auto-renders by mapping `COLUMN_KEYS` → i18n `list.columns.<key>` in page.tsx; add that message key too.
- 2026-06-17 · codebase-pattern · SUPERSEDES the `?severity` filter-bar entry above: the severity-findings feature shipped as HOVER POPOVERS on two surfaces, NOT a PR-wide filter bar. The filter-bar approach (`SeverityFilterBar`, `?severity` URL param, `visibleFindings` severity threading) was reverted — those files no longer exist; ignore that entry's specifics. Live surfaces: (1) PR-list FINDINGS column — `_components/PRRow/PRRow.tsx` renders `FindingsSummary` in a `HoverCard` whose body is the lazy `_components/PRRow/PrFindingsHover.tsx`; (2) detail timeline per-run counts — `RunHistory.tsx` reads a `findingsByRun: Map<run_id, FindingRecord[]>` built in `FindingsTab.tsx` (keyed `ReviewRecord.run_id` ↔ `RunSummary.run_id`; falls back to the old text count when a run_id is absent). Shared, reusable pieces in `client/src/components/`: `hover-card.tsx`, `findings-summary.tsx` (compact severity badges, CRITICAL→WARNING→SUGGESTION, skips zero counts → null when all zero), `findings-popover.tsx` (read-only list, no accept/dismiss actions).
- 2026-06-17 · codebase-pattern · PR-list findings badge counts come from a server field `findings_summary: {CRITICAL,WARNING,SUGGESTION}|null` on `PrMeta` (derived from each PR's LATEST review). The hover popover then lazy-fetches the full findings via `usePrReviews(pr.id)` and uses `reviews[0]` (reviews are server-ordered `desc(createdAt)`, so [0] = latest = the same review the badges counted → totals match). The lazy fetch is gated by `HoverCard` mounting `children` ONLY while open — preserve that contract if you touch HoverCard, or every row would fetch eagerly.
- 2026-06-18 · codebase-pattern · SUPERSEDES the "hover-only, NOT click-to-filter" qualifier in the earlier severity-findings entry: the badges are now CLICK-TO-FILTER on BOTH surfaces (PR list + detail timeline). `FindingsSummary` ({counts, onSelect?, activeSeverity?}) renders each badge as a `<button>` ONLY when `onSelect` is passed (omit it → plain badges, unchanged); `SeverityBadge` is vendored/do-not-touch so the click lives on the wrapping button, with non-active badges dimmed (`opacity .4`) + `aria-pressed`. Behaviour: click a severity → filter the popover to it; click it again → clear (toggle); the filter RESETS when the popover closes, via a new `HoverCard` `onClose?` prop (fires on open→closed). Each surface holds its own `activeSeverity` useState and filters the findings BEFORE passing to `FindingsPopover`. KEY: where rows render in a `.map` (the timeline in `RunHistory.tsx`), the state+HoverCard MUST be extracted into a child component (`RunFindings`) — you can't `useState` per-row inside the map. The PR list already had a per-row component (`PRRow` / `PrFindingsHover`). Test the timeline filter by selecting badges via `aria-pressed` (the row's other buttons — agent-name, trace — lack it) and open the HoverCard by `fireEvent.mouseEnter` on `badge.parentElement.parentElement` (mouseenter doesn't bubble; that's the HoverCard trigger span).

## Tool & Library Notes
<!-- dependency quirks/gotchas -->
- 2026-06-17 · gotcha · Running `pnpm build` in `client/` while a stack is already up (`./scripts/dev.sh` runs `next-server` on :3000) overwrites the shared `.next` dir and corrupts the running server's chunk graph — every route that pulls a vendor chunk then 500s with `Cannot find module './vendor-chunks/recharts.js'` (or similar). It is NOT a code bug; restarting the dev server clears it. To verify a change against the live stack without nuking it, DON'T `pnpm build` alongside it — rely on `pnpm typecheck` + `pnpm test`, or restart the server after building. Symptom check: if `/repos/.../pulls/<n>` 500s but `/repos/.../pulls` (list, no charts) is 200, suspect this, not your diff.
