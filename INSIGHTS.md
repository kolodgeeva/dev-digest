- 2026-06-27 ┬╖ codebase-pattern ┬╖ Two new subagents added: `.claude/agents/planner.md` (model opus, read-only + writes only `.claude/plans/<slug>.md`) and `.claude/agents/implementer.md` (model sonnet, edits code). Intended flow: planner emits a Development Plan with a **File ownership** section that partitions work into NON-OVERLAPPING file sets (typically `server/**` vs `client/**`) тЖТ multiple `implementer`s run in parallel **on the same branch** (no worktree isolation, by request) and each stays inside its slice's files. Implementer self-reviews only its own diff + runs `pnpm typecheck` and unit tests (`--exclude '**/*.it.test.ts'`); it is NOT the quality gate тАФ `pr-self-review` still runs at commit. If you change the parallelism model, the non-overlap invariant is what prevents same-branch write collisions.

## Tool & Library Notes
<!-- dependency quirks/gotchas -->
- 2026-06-21 ┬╖ tooling ┬╖ Root `CLAUDE.md` is a symlink to `AGENTS.md` (`readlink -f CLAUDE.md` тЖТ `AGENTS.md`). The Edit/Write tools refuse to write through a symlink тАФ edit `AGENTS.md` directly. (Per-package `server/CLAUDE.md` etc. are real files.)
- 2026-06-21 ┬╖ tooling ┬╖ New `pr-self-review` skill + commit gate added this session. `git commit` is blocked by a `PreToolUse` hook in `.claude/settings.json` until a pass flag `/tmp/claude-pr-selfreview-<diffhash>` exists; the skill (`/pr-self-review`) writes that flag only when 0 CRITICAL findings. Flag is keyed to a hash of `git diff $(git merge-base main HEAD)` тАФ NOT session id тАФ because the hook reads `.session_id` from stdin JSON while the skill's Bash env has no reliable matching session var; diff-hash keying also forces re-review whenever code changes. The skill routes changed files to existing skills by path (UIтЖТfrontend-architecture/next/react, serverтЖТonion/fastify/drizzle/pg, allтЖТtypescript-expert/zod/security). To bypass during an authorized commit, create the flag for the current diff hash.
- 2026-06-21 ┬╖ tooling ┬╖ CORRECTION to the `pr-self-review` note above. Two changes after first use: (1) **Scope is now local changes vs the current branch tip `HEAD`** (staged + unstaged + untracked), NOT vs `main` тАФ committed branch work isn't re-reviewed. (2) **The fingerprint is computed through a throwaway index**, not `git diff <merge-base>`. The old hash excluded untracked files, so a new file slipped past review AND `git add`-ing it shifted the hash тЖТ guaranteed one "blocked" cycle on every new file. Fix (same in both skill Step 1/5 and the hook): `tmp=$(mktemp); GIT_INDEX_FILE="$tmp" git read-tree HEAD; GIT_INDEX_FILE="$tmp" git add -A; h=$(GIT_INDEX_FILE="$tmp" git diff --cached HEAD | shasum | cut -d' ' -f1); rm -f "$tmp"`. This is staging-invariant (verified) and includes untracked files. The real index is never touched (temp `GIT_INDEX_FILE`).

- 2026-06-27 ┬╖ tooling ┬╖ 2026 research to bake into any test-generating agent's prompt as an explicit forbid-list: AI agents systematically **over-mock** (arxiv 2602.00409 тАФ 36% of agent test commits add mocks vs 26% human; agents use mocks for ~95% of doubles vs humans' mix of fakes/spies) and write **tautological assertions** (assert a mock returns what you told it to). The `test-writer` agent body therefore hard-FORBIDS: mocking the subject under test / your own modules/hooks/context, snapshot tests, `fireEvent` (use `userEvent.setup()`), `container.querySelector` (use `screen` role/label queries), importing from `jest`, asserting `useState`/internal hook state, and tautological assertions. Mock only at boundaries (network/adapters/external SDKs). Spec: `.claude/plans/new-agents.md` ┬з6.1.
- 2026-06-27 ┬╖ tooling ┬╖ A subagent's `skills:` frontmatter list (used in `.claude/agents/{planner,implementer}.md`) **preloads the full content of each listed skill into the agent's context at startup** тАФ it is NOT lazy. Both agents list the full backend+UI+cross-cutting skill set; planner additionally lists `mermaid-diagram`. Rationale: planner plans the implementation and assigns skills per task, so it needs the full content of every skill the implementer will use, not just the names тАФ by design the planner's skill set is a superset of the implementer's. This is separate from on-demand `Skill` tool invocation, which both agents also use for the routing-table lookups.
